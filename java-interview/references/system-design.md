## 分布式理论

### CAP 定理（高频⭐⭐⭐⭐）

- **C（Consistency）一致性**：所有节点同一时刻看到的数据相同
- **A（Availability）可用性**：每个请求都能收到响应（不保证最新数据）
- **P（Partition Tolerance）分区容错性**：网络分区时系统仍能运行

**结论**：网络分区不可避免，P 必须保证，只能在 C 和 A 之间权衡

- **CP**（如 ZooKeeper、HBase）：保证强一致，网络问题时不可用
- **AP**（如 Eureka、Cassandra）：保证可用，允许短暂数据不一致

### BASE 理论

对 CAP 中 AP 方案的补充：
- **Basically Available（基本可用）**：核心功能可用，允许局部降级
- **Soft State（软状态）**：允许中间状态，不要求时刻一致
- **Eventually Consistent（最终一致性）**：经过一段时间后最终一致

## 微服务

### 服务注册与发现

| 组件 | 一致性 | 健康检查 | 特点 |
|------|--------|---------|------|
| Eureka | AP | 心跳 | 自我保护，不轻易删除节点 |
| Nacos | AP+CP | 主动探测 | 同时支持注册中心和配置中心 |
| ZooKeeper | CP | Session 超时 | 强一致，适合协调服务 |
| Consul | CP | 多种方式 | 支持多数据中心 |

### 负载均衡策略

- **轮询（Round Robin）**：依次分发，适合服务器性能均等
- **加权轮询**：按权重分发，适合性能差异大
- **随机**：随机选择
- **最少连接**：选择当前连接数最少的服务器
- **一致性哈希**：相同 key 路由到同一节点（适合缓存场景）

## 限流与熔断

### 限流算法（高频⭐⭐⭐⭐）

**固定窗口**：单位时间内计数，超阈值拒绝
- 缺点：窗口边界可能瞬间通过 2 倍流量

**滑动窗口**：将窗口细分，统计滑动区间内的请求数
- 修复固定窗口边界问题

**漏桶算法**：请求以固定速率流出（消费），超出桶容量则丢弃
- 优：平滑输出；缺：无法应对突发流量（无弹性）

**令牌桶算法（推荐）**：匀速向桶中放令牌，请求需消耗令牌
- 优：允许一定突发（桶中积累的令牌）
- Guava `RateLimiter` 实现令牌桶

```java
RateLimiter limiter = RateLimiter.create(100); // 每秒100个令牌
if (limiter.tryAcquire()) {
    // 处理请求
} else {
    // 限流降级
}
```

### 熔断器（Circuit Breaker）

**状态机**：
```
CLOSED（正常）→ OPEN（熔断）→ HALF_OPEN（半开）→ CLOSED（恢复）
```

- **CLOSED**：正常请求，统计失败率
- **OPEN**：失败率超阈值，直接拒绝请求（快速失败）
- **HALF_OPEN**：超过等待时间，放少量请求探测，成功则关闭熔断

**Hystrix vs Resilience4j**：
- Hystrix（Netflix，已停止维护）：线程池隔离
- Resilience4j（推荐）：更轻量，函数式 API，支持 Reactor/RxJava

## 分布式事务（高频⭐⭐⭐⭐）

### 常见方案

**2PC（两阶段提交）**：
1. 准备阶段：协调者询问各参与者是否可提交
2. 提交阶段：所有同意则提交，否则回滚

缺点：同步阻塞、单点问题（协调者挂了）、数据不一致风险

**TCC（Try-Confirm-Cancel）**：
- Try：预留资源（冻结库存、冻结金额）
- Confirm：确认提交（真正扣减）
- Cancel：取消释放（解冻）

特点：业务侵入强，但灵活、高性能

**Saga（长事务）**：
- 将长事务拆分为多个本地事务
- 每步失败时执行对应补偿操作（反向操作）
- 适合长流程业务（如订单→支付→库存→物流）

**本地消息表（推荐，简单可靠）**：
1. 业务操作 + 写消息表（同一本地事务）
2. 异步发送消息到 MQ
3. 消费者消费并更新状态
4. 重试机制保证最终一致

**Seata**：阿里开源，支持 AT（自动补偿）/TCC/Saga 模式

## 高可用设计

### 常见手段

**冗余**：主从备份、多副本（避免单点故障）

**隔离**：
- 线程池隔离：不同业务用不同线程池（Hystrix）
- 进程隔离：微服务本身就是隔离
- 机房隔离：同城双活、异地多活

**降级**：核心功能不受影响，非核心功能降级处理
```java
// 例：推荐服务挂了，返回热门列表兜底
try {
    return recommendService.getForUser(userId);
} catch (Exception e) {
    return hotListService.getHotList();
}
```

**超时与重试**：
- 设置合理超时（防止资源耗尽）
- 重试需保证幂等
- 指数退避（Exponential Backoff）+ Jitter（随机抖动）

### 数据库高可用

**主从复制**：主库写，从库读（读写分离）
- 异步复制（默认）：性能好，可能丢数据
- 半同步复制：至少一个从库确认，平衡性能与安全

**主从延迟问题**：
- 写后立即读走主库
- 关键操作强制读主库
- 等待复制完成（特定场景）

**分库分表**：
- 垂直分库：按业务模块拆分
- 水平分表：按规则（用户 ID 取模）拆分数据
- 中间件：ShardingSphere、MyCat
