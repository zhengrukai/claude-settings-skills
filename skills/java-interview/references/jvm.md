## 内存结构

### JVM 运行时数据区（高频⭐⭐⭐⭐⭐）

```
┌─────────────────────────────────────────────┐
│                  所有线程共享                  │
│  ┌──────────────────┐  ┌──────────────────┐  │
│  │    堆（Heap）     │  │   方法区（元空间）  │  │
│  │  对象实例 / 数组  │  │  类信息/常量/静态  │  │
│  └──────────────────┘  └──────────────────┘  │
├─────────────────────────────────────────────┤
│                  线程私有                      │
│  ┌──────┐  ┌────────────┐  ┌─────────────┐  │
│  │程序计数│  │  虚拟机栈  │  │  本地方法栈  │  │
│  │  器  │  │ 栈帧/局部变量│  │ Native方法  │  │
│  └──────┘  └────────────┘  └─────────────┘  │
└─────────────────────────────────────────────┘
```

**各区域详解**：

| 区域 | 存储内容 | 异常 | 是否线程共享 |
|------|---------|------|------------|
| 程序计数器 | 当前字节码行号 | 无（唯一）| 否 |
| 虚拟机栈 | 栈帧（局部变量表、操作数栈、返回地址）| StackOverflowError / OOM | 否 |
| 本地方法栈 | Native 方法信息 | StackOverflowError / OOM | 否 |
| 堆 | 对象实例、数组 | OOM | 是 |
| 方法区（元空间）| 类元信息、常量池、静态变量 | OOM | 是 |

**JDK 8 方法区变化**：永久代（PermGen）→ 元空间（Metaspace）
- 元空间使用本地内存，不再受 JVM 堆大小限制
- 默认不设上限（可通过 `-XX:MaxMetaspaceSize` 限制）

### 堆内存结构

```
堆（Heap）
├── 新生代（1/3）
│   ├── Eden（8/10）
│   ├── Survivor From（1/10）
│   └── Survivor To（1/10）
└── 老年代（2/3）
```

**对象晋升老年代的条件**：
1. 年龄计数器达到阈值（默认 15，`-XX:MaxTenuringThreshold`）
2. Survivor 区放不下的大对象
3. 动态年龄判断：相同年龄对象总大小超过 Survivor 一半

## 垃圾回收

### 判断对象是否可回收

**引用计数法**（Java 不使用，无法解决循环引用）

**可达性分析（GC Roots）**：
GC Roots 包括：
- 虚拟机栈中局部变量引用的对象
- 方法区中静态变量引用的对象
- 方法区中常量引用的对象
- 本地方法栈 JNI 引用的对象
- 被同步锁（synchronized）持有的对象

### 四种引用类型

| 引用类型 | 回收时机 | 典型用途 |
|---------|---------|---------|
| 强引用 | 永不回收（OOM 也不回收）| 普通 `new` 对象 |
| 软引用 | 内存不足时回收 | 缓存 |
| 弱引用 | 下次 GC 必回收 | WeakHashMap、ThreadLocal |
| 虚引用 | 随时可回收，收到通知 | 堆外内存管理 |

### 垃圾收集算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记可回收对象后清除 | 简单 | 内存碎片 |
| 标记-整理 | 清除后将存活对象整理到一端 | 无碎片 | 移动耗时 |
| 复制算法 | 将存活对象复制到另一半空间 | 快、无碎片 | 浪费一半内存 |
| 分代收集 | 新生代用复制，老年代用标记-整理 | 综合最优 | - |

### 垃圾收集器（高频⭐⭐⭐⭐）

| 收集器 | 适用区域 | 算法 | 特点 |
|-------|---------|------|------|
| Serial | 新生代 | 复制 | 单线程，STW |
| ParNew | 新生代 | 复制 | 多线程版 Serial |
| Parallel Scavenge | 新生代 | 复制 | 吞吐量优先 |
| CMS | 老年代 | 标记-清除 | 低停顿，并发 |
| G1 | 全堆 | 分区复制+标记整理 | 可预测停顿时间 |
| ZGC | 全堆 | 染色指针 | 停顿 < 10ms（JDK 11+）|

**CMS 四个阶段**：
1. 初始标记（STW）：标记 GC Roots 直接关联对象
2. 并发标记：追踪引用链（与用户线程并发）
3. 重新标记（STW）：修正并发期间变动
4. 并发清除：清理垃圾（与用户线程并发）

**CMS 缺点**：内存碎片（标记-清除）、浮动垃圾、CPU 敏感

**G1 特点**：
- 将堆划分为等大的 Region（1~32MB）
- 不分固定新生代/老年代，Region 角色动态变化
- 维护每个 Region 的回收价值，优先回收价值大的（Garbage First）
- 可设置 GC 停顿目标：`-XX:MaxGCPauseMillis=200`

## 类加载机制

### 类加载流程

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
          └────────连接──────────┘
```

- **加载**：读取 .class 字节码，创建 Class 对象
- **验证**：字节码合法性检查
- **准备**：为静态变量分配内存，赋**默认**初始值（int → 0）
- **解析**：符号引用 → 直接引用
- **初始化**：执行 `<clinit>()` 静态块和静态变量赋值

### 类加载器双亲委派（高频⭐⭐⭐⭐）

```
Bootstrap ClassLoader（启动类加载器）
    ↑
Extension ClassLoader（扩展类加载器）
    ↑
Application ClassLoader（应用类加载器）
    ↑
自定义 ClassLoader
```

**流程**：先委托父类加载，父类无法加载时才自己加载

**作用**：
- 防止核心类库被替换（如自定义 `java.lang.String` 无效）
- 保证类的唯一性

**打破双亲委派的场景**：
1. JNDI、JDBC 等 SPI 机制（使用线程上下文类加载器）
2. OSGi 模块化框架
3. Tomcat 为每个 WebApp 使用独立类加载器

## JVM 调优

### 常用 JVM 参数

```bash
# 堆大小
-Xms512m -Xmx2g

# 新生代大小
-Xmn512m

# 元空间
-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m

# 选择 GC
-XX:+UseG1GC
-XX:+UseConcMarkSweepGC
-XX:+UseZGC  # JDK 11+

# GC 日志
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/logs/gc.log

# OOM 时 dump
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof
```

### 常见 OOM 场景及排查

| OOM 类型 | 原因 | 排查工具 |
|---------|------|---------|
| Java heap space | 对象无法回收、内存泄漏 | jmap + MAT |
| Metaspace | 类加载过多（动态代理/反射）| jstat -gcmetacapacity |
| GC overhead limit exceeded | GC 耗时超过 98%，回收不足 2% | GC 日志 |
| Direct buffer memory | 堆外内存泄漏（NIO/Netty）| `-XX:MaxDirectMemorySize` |
| StackOverflowError | 递归调用过深 | 栈帧分析 |

### 常用诊断工具

```bash
jps          # 查看 Java 进程
jstack <pid> # 线程堆栈（排查死锁/CPU 100%）
jmap -heap <pid>   # 堆使用情况
jmap -dump:format=b,file=heap.hprof <pid>  # 导出堆快照
jstat -gcutil <pid> 1000  # GC 统计（每秒输出）
```

**CPU 100% 排查步骤**：
1. `top` 找到高 CPU 的 Java 进程 PID
2. `top -H -p <pid>` 找到高 CPU 的线程 TID
3. `printf '%x\n' <TID>` 转为十六进制
4. `jstack <pid> | grep <hex_tid>` 找到线程堆栈
